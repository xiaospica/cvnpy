# vnpy_signal_strategy_plus 实盘异常自动化测试说明

## 背景

`vnpy_signal_strategy_plus` 的核心链路是：UI启动策略 → 策略轮询 MySQL 表 `stock_trade` → 解析信号 → 通过网关下单 → 接收委托/成交回报 →（可能触发）网关超时撤单与策略层撤单重挂。

实盘中常见风险点集中在订单生命周期异常：拒单、未成交、部分成交、撤单失败、回报延迟/乱序、断连重连后状态收敛不一致等。本测试方案的目标是在不引入复杂外部测试框架的情况下，用“写入测试信号”驱动真实下单链路，尽可能覆盖上述异常处理逻辑。

## 重要安全提示

- 强烈建议使用模拟环境或小资金账户执行。
- 测试用例默认使用小比例仓位，但仍可能成交并产生真实盈亏。
- 执行前确保已开启：网关超时撤单（如 `vnpy_qmt` 或 `vnpy_xt` 的超时撤单功能），并且策略可随时停止。
- UI 中只有选择“测试策略”时才会启用自动化测试按钮，避免误操作影响正常策略。

## 关于测试标的的选择（T+0 vs T+1）

测试套件需要在**同一交易日内**完整跑通 buy → sell 链路，因此**必须使用二级市场 T+0 的标的**，否则当日 buy 成交后形成的持仓被 T+1 冻结，sell_smoke / sell_passive 会被策略侧"未找到可卖持仓"拦截，无法验证卖出路径。

A 股 ETF T+0/T+1 速查（截至 2026 年规则）：

| 类别 | 例子 | 二级市场 |
|---|---|---|
| 黄金 ETF（推荐） | **518880.SH**（华安黄金）, 518800.SH, 518680.SH | **T+0** |
| 跨境 ETF（QDII） | 513100.SH（纳指）, 513050.SH（中概互联）, 513500.SH（标普 500）, 159920.SZ（恒生） | **T+0** |
| 商品期货 ETF | 159980.SZ（有色）, 159985.SZ（豆粕） | **T+0** |
| 债券 ETF | 511010.SH, 511260.SH | **T+0** |
| 货币 ETF | 511880.SH, 511990.SH | T+0（价格恒为 100，无涨跌停意义，不适合本测试） |
| 普通股票 ETF | 510300.SH（沪深 300）, 510050.SH, 510500.SH, 159928.SZ（消费） | **T+1**（不推荐） |

默认 `test_symbol = "518880.SH"`（黄金 ETF）：

- 流动性高（日均成交几十亿）
- 有 ±10% 涨跌停（可触发 reject_up / reject_down）
- 价格档位正常（pricetick=0.001，价格区间 5–8 元）
- 国内最主流、最稳定的 T+0 ETF

**如果必须使用 T+1 标的**（例如要测特定行业 ETF 链路），有两条变通路径：
1. 测试前**不执行一键清仓**，靠隔夜底仓的可卖部分支撑 sell_*。
2. 跨日两段跑：D1 只跑 buy_*，D2 再跑 sell_*。

## 如何运行

1. 在“信号策略Plus”界面添加并启动测试策略：
   - 策略类：`LiveOrderTestStrategyPlus`
   - 策略名：`live_order_test`
2. 确保测试策略已成功连接数据库（日志出现“数据库连接成功”）。
3. 在策略下拉框选中 `live_order_test`，点击“自动化测试”按钮：
   - 会先弹出风险确认
   - 然后选择套件：全部/冒烟/基础/全量
4. UI 会向 MySQL 表 `stock_trade` 写入一组测试信号；策略轮询线程读取后自动下单。
5. 测试信号写入后会在日志中打印每条 signal 的 id/code/pct/type，便于你事后把 log 发回我定位问题。

## 用例分层与覆盖点

### 1) 冒烟（Smoke）

目标：用最短时间验证“能下单、能回报、能撤单/超时撤单”基本链路。

信号序列（同一标的，默认 `518880.SH`（黄金 ETF，T+0），仓位比例 `test_pct`）：
- `buy_smoke_<run_id> aggressive`
  - 期望：正常发单；更倾向于快速成交（取价使用买一/卖一）。
  - 覆盖：信号写库→轮询→下单→委托回报/成交回报基础闭环。
- `sell_smoke_<run_id> aggressive`
  - 期望（默认 T+0 标的）：上一条 buy_smoke 成交后形成可用持仓，sell_smoke 正常下卖并成交。
  - 期望（T+1 标的）：当日买入被冻结，可用为 0，会被策略侧拦截（日志提示“未找到可卖持仓”）；若改用隔夜底仓则正常卖出。
  - 覆盖：卖出链路、持仓保护逻辑、基础回报处理。

### 2) 基础（Basic）

目标：覆盖核心异常类别：撤单重挂、拒单、非最优价挂单、无持仓强制卖出拒单、非法数量拒单。

**用例顺序设计原则**（重要，避免时序冲突导致语义破坏）：
1. `force_sell_no_position` 必须放在序列最前面，**只有此时账户中该标的可卖持仓 = 0**，才能验证"无持仓→[251005] 拒单"语义；放在 buy 用例之后会因为账户已积累持仓而被柜台撮合成交。
2. 通过双倍仓位 `buy_smoke_double`（pct\*2）建立底仓，紧跟 `sell_smoke`（pct，卖一份），让后续 `sell_passive` 时账户里至少还留一份可卖部分；这样 sell_passive 不依赖排队中的 `buy_passive` 是否成交，能稳定走到 passive 排队链路。
3. `reject_up` / `invalid_volume` 即使触发或被拒，对账户持仓影响可控（前者本应被拒；模拟柜台下即使错误成交，多余持仓也不破坏其它用例语义；后者只下 1 股拒单，持仓不变），放在序列后段。

信号序列（按上述顺序）：

1. `sell_force_no_pos_<run_id> force_sell_no_position`
   - 期望：绕过策略侧持仓校验强制下 100 股卖单到柜台，在无持仓时被柜台拒单（实盘常见错误码 [251005] "证券可用数量不足"）。
   - 前置约束：账户当前该标的可卖持仓 = 0（依赖一键清仓 + 此用例最先执行）。

2. `buy_smoke_double_<run_id> buy_smoke_double aggressive`
   - 期望：以 `pct*2` 仓位 aggressive 定价（卖一价）快速成交，建立底仓。
   - 覆盖：信号写库→轮询→下单→委托回报/成交回报基础闭环 + 为后续卖出用例提供可卖持仓。

3. `sell_smoke_<run_id> aggressive`
   - 期望：以 `pct` 仓位 aggressive 定价（买一价）卖出一份，账户保留 ≈ pct 仓位的可卖部分。

4. `buy_passive_<run_id> passive`
   - 期望：更偏向排队挂单；若启用网关超时撤单，则可能触发撤单与策略重挂。

5. `sell_passive_<run_id> passive`
   - 期望：依赖用例 2/3 留下的可卖底仓（不依赖用例 4 是否成交），稳定进入 passive 排队链路；若启用网关超时撤单，可能触发撤单 + 策略重挂。
   - T+1 标的注意：当日 buy 形成的持仓被 T+1 冻结，本用例会被策略侧"未找到可卖持仓"拦截，无法验证 passive 排队。要真正验证此路径需用 T+0 标的（默认）或保留隔夜底仓。

6. `buy_reject_up_<run_id> reject_up`
   - 期望：构造高于涨停价的买入价格，触发拒单。
   - 实现说明：测试策略先把 `limit_up * 1.05` 按 `pricetick` 对齐，确保不会先因"最小价差校验 [120161]"被拒单，而是命中真正的"价格越界"路径。
   - **柜台行为差异**：实盘严格柜台（不同券商策略不同）会返回 [120160]/[251006] 类越界拒单；**模拟柜台（QMT_SIM 或 QMT 模拟账户）通常不校验越界，会按盘口价撮合成交**——此情形下视为"该用例在当前柜台环境下不必现拒单"，不视为失败。建议在真实柜台环境验证此用例。

7. `buy_no_fill_60s_<run_id> no_fill_60s passive`
   - 期望：
     - `QMT_SIM`：根据 `|case=no_fill_60s` 保持不成交，等待超时撤单触发 `CANCELLED`，并观察策略重挂。
     - `QMT` 实盘：不保证必现"不成交"，但可用于观察"未成交→超时撤单→重挂"的实盘链路与日志完整性。

8. `buy_invalid_volume_<run_id> invalid_volume`
   - 期望：构造"委托数量不合法"拒单，用于验证柜台拒单回报与日志口径（低风险、可控）。
   - 实现说明：测试策略绕过 `MySQLSignalStrategyPlus.send_order` 的"数量修正/拦截"，直接通过 `SignalTemplatePlus.send_order` 把 1 股下到柜台，触发实盘柜台层 [120155]/[120158] 类拒单。

### 3) 全量（Full）

目标：扩大覆盖面（更多异常触发点），用于发版前回归。

说明：
- `QMT_SIM`：会额外启用"延迟成交/部分成交后停滞/强制拒单/跌停越界拒单"等撮合用例。
- `QMT` 实盘：为避免不可控风险与不确定性，"全量"沿用"基础"序列（自包含 8 条；不再叠加冒烟，避免冒烟里的 sell_smoke 卖光持仓打乱后续 sell_passive / force_sell_no_pos 的时序约束）。

其中：
- `passive/deep` 用于制造未成交/长时间排队，从而触发超时撤单与重挂逻辑。
- `reject_up/reject_down` 用于构造价格越界拒单路径（能否必现取决于券商校验与交易时段）。

## 信号字段约定

写入 MySQL 表 `stock_trade` 的关键字段：
- `stg`：固定为 `live_order_test`（用于策略过滤）
- `code`：默认 `518880.SH`（内部会转换为 vn.py 的 vt_symbol；选择 T+0 标的的原因见上文"测试标的的选择"章节）
- `pct`：仓位比例（必须 `<= 1.0`，否则当前策略会直接报“百分比异常”并忽略）
- `type`：包含 buy/sell 关键字，并可追加标签：
  - `aggressive`：用更积极的盘口价格挂单（更倾向于快速成交）
  - `passive`：用更保守的盘口价格挂单（更倾向于不成交/排队）
  - `invalid_volume`：强制下发非法数量（如 1 股）用于拒单链路验证
  - `reject_up` / `reject_down`：构造价格越界拒单
  - 仅 `QMT_SIM`：当 `type` 中包含 `no_fill_60s/delayed_fill_5s/partial_then_stall_5s/force_reject/force_sell_no_position` 等 token 时，会通过 `OrderRequest.reference` 注入 `|case=...` 以驱动模拟撮合行为
